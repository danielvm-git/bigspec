Where the idea came from

bigpowers proved the thesis. It also taught the lesson.

I spent 41 days building bigpowers — a prescriptive methodology that distills 17 years of software-engineering discipline into agent skills. It worked. 98 releases, 98% compliance, a real lifecycle, a live cockpit. But somewhere around the 74th skill I realised I was maintaining complexity I had created, not complexity the problem demanded.

So I stopped adding and started listening to the repo. It kept telling me the same thing: most of what I built was accidental. The skill-count was vanity. The 27 reference docs restated one constitution. The four-target sync pipeline was the single biggest source of duplication. Three different things were all wearing the word "skill."

bigspec is the version I'd build if I knew all that on day one — which now I do. Same soul, none of the debt, poured in the right order. This page is me laying out the whole foundation before I pour a single block, so I don't repeat myself.

How the ideas concatenate

Not a flat list of influences — a layer cake.

Each wave of thinking builds on the last and resolves a tension the previous one created. No layer replaces its predecessor; it answers a problem the predecessor opened. This lineage is the intellectual spine bigspec inherits.

2008
Uncle Bob — Clean Code
SRP, Boy Scout Rule, F.I.R.S.T tests, intention-revealing names.
The craftsmanship floor. Everything stands on it.
2018
Ousterhout — A Philosophy of Software Design
Deep modules, information hiding, define errors out of existence.
Resolves: small functions alone create shallow modules with bloated interfaces.
2023–24
Karpathy · Pocock · Superpowers
Think-first planning, verb-noun skill architecture, hard gates, zoom-out strategy.
Resolves: raw LLMs have no discipline — they need orchestration, not raw prompting.
2024
Wasowski (SDD) · BCP
Specs as the human–agent interface; business complexity as a pre-build sizing unit.
Resolves: agents drift without a verifiable spec — BDD Gherkin closes the loop.
2026
Akita — Clean Code for AI Agents · GSD · BMAD · OKF
Grep-ability & token economy · context-rot & fresh subagents · document lifecycle & P0–P3 risk tiers · one universal artifact envelope.
Resolves: rules written for humans need re-ranking for agents; context rot is the real failure mode.

Building without the same errors

Every bigpowers primitive, put on trial.

A blank page is only valuable if you know what survives contact with it. Each load-bearing idea was cross-examined: keep it, cut it, or replace it. This board is the difference between a rewrite and a rebuild.

KEEP the soul
  • Spec as source of truth — survives code regeneration
  • The layer-cake doctrine — provenance in one place
  • Verb-noun procedures — a public API of skills
  • The fractal loop — one shape at every scope
  • File-based YAML cockpit — diffable, greppable, no server
  • BCP sizing — "how much effort is left?"
  • next_skill handoff — resume after any interruption
  • semantic-release — versioning is automated truth
CUT the vanity
  • 74-skill count as a feature — a metric that fights maintenance
  • 27 reference docs — they just restate the constitution
  • Bespoke 4-target sync pipeline — the biggest source of duplication
  • Dual epic formats — one capsule directory, always
  • Skills for tools & rules — those are kernel and constitution
  • _LATEST filename hack — git already versions and freezes
REPLACE the shape
  • 6-phase lifecycle → 5 fractal movements, spec-anchored
  • 94% self-compliance gate → outcome evals (with/without-skill Δ)
  • Uniform hard gates → risk-tiered P0–P3 (fast lane vs full lane)
  • Skills-first, infra-late → kernel-first, MCP-last
  • External counting tool → native count-bcp

How the agent thinks
B0
Token / Context Substrate
Everything is authored under a token budget — written to be consumed linearly by a machine.
▾
RuleProgressive disclosure (frontmatter → summary → detail); no filler or restated code; one-line runnable verify: commands; deep modules over chatty ones.
ProvenanceContext Engineering — Write / Select / Compress / Isolate.
B1
Agent-Grade Craftsmanship
The code floor re-ranked for agents: SRP, deep modules, grep-able symbols, remediation-carrying errors.
▾
RuleFunctions do one thing (4–20 lines); files < 300 lines; deep modules (simple interface, hidden complexity); public symbols grep to < 5 results; errors carry the offending value + a remediation hint.
ProvenanceUncle Bob (Clean Code) → Ousterhout (deep modules) → Akita (Clean Code for AI Agents).
B2
Think-First Behavior
Understand before acting; surface assumptions; ship the minimum; loop until verified, not assumed.
▾
RuleSurface assumptions before writing code; present multiple readings of an ambiguous request; ship the MVP; edits are surgical; loop until behavioral correctness is verified.
ProvenanceKarpathy; extended-thinking / reasoning patterns.
How work is structured
B3
The Skill Primitive
A skill is only a procedure. Rules live in the constitution; tools live in the kernel.
▾
RuleVerb-noun kebab-case names (a public API); every skill declares effort, spawn, risk, movement and a behavioral verify:; each writes handoff.next_skill as its last action.
ProvenancePocock (Agent Skills); domain probes.
B4
Spec-as-Contract
The durable, testable spec is the source of truth — not the chat, not the code. Code is last-mile output.
▾
RuleEvery epic has a spec with scope, ubiquitous-language glossary, and Gherkin acceptance criteria; the spec feeds the story→BCP pipeline; every criterion is a verifiable assertion with a runnable verify:.
ProvenanceWasowski (SDD + BDD); Spec-Kit (constitution + executable intent).
How quality is enforced
B5
Risk-Tiered Verification
Proof is proportionate to risk. Trivial work takes the fast lane; critical work takes the full gauntlet.
▾
RuleEvery story is tagged P0–P3 at plan time; verification depth follows the tier; tests obey F.I.R.S.T and assert behavior through public APIs; NFRs are gates for P0/P1, not afterthoughts.
ProvenanceBMAD TEA (P0–P3 risk-based test depth, NFR gates); TDD / F.I.R.S.T; STRIDE.
B6
Gates & Hard-Stops
Progress blocks until conditions are met — but gates are risk-scaled, so they never become fatigue.
▾
RuleNo code before the design is approved; name "red-flag" rationalizations ("too simple to plan"); independent reviews run in a fresh subagent context; two-stage review (self-audit → external) precedes merge.
ProvenanceSuperpowers (hard gates, red flags); four gate types — Confirm / Quality / Risk / Completeness.
How context is managed
B7
Context Isolation
Context rot is the primary failure mode. The fix is fresh-context subagents with file-based handoff.
▾
RuleHeavy work (effort: heavy, spawn: subagent) runs in a fresh context and returns only its result; the orchestrator stays lean; handoff.next_skill lets any interruption resume exactly where it stopped.
ProvenanceGSD (context-rot thesis, fresh-context subagents, effort budgeting).
B8
Self-Describing Artifacts (OKF)
Every generated file is an OKF bundle. One envelope, one validator, templates as data.
▾
RuleEach artifact carries YAML frontmatter with okf_kind, okf_version, provenance; validate-okf is the single kind-aware gate for specs/; validators gate on structure & provenance, never a value. _LATEST is abolished.
ProvenanceOKF — Open Knowledge Format. See the spine below ↓
How effort is tracked
B9
Effort Accounting (BCP)
One sizing unit runs the roadmap — counted from the spec by native count-bcp, never guessed.
▾
RuleBCP is a sum of per-dimension element sizes, counted from the story by an Element Router; stories carry no gestalt SIZE; every count is gated on calibration_id, never the number; velocity series segment by calibration. Anchored to a known-3 and known-8 exemplar.
ProvenanceBCP — Business Complexity Points (CI&T / Itaú); BCP-Plus 13-dimension methodology.
How it all coheres
B10
The Synthesis
One fractal loop, one constitution, hard gates scaled by risk — the principles become one discipline.
▾
RuleThe loop (Frame → Specify → Plan → Build → Prove → ship) runs identically at project, epic, and story scope; every skill maps to exactly one movement; the spec is the invariant across every movement; the constitution is one file.
ProvenanceBMAD (document lifecycle) + GSD (execution framework), synthesized.

★
Capstone · Outcome Evals
The foundation proves it produces better software — a with-skill vs without-skill delta on real tasks — not that it obeys its own philosophy.

The whole vocabulary

Seven nouns. That's the entire model.

A blank repo needs a small, orthogonal vocabulary. The discipline that keeps the corpus from ballooning back to 74: rules are not skills, tools are not skills, docs are not skills.

Constitution
One file. Immutable principles, risk tiers, conventions. The single source of rules.
Spec
The durable, testable statement of intent. Lives in git; survives code regeneration.
Cockpit
Machine state — active epic, handoff, metrics. YAML files via bp-yaml. No server.
Skill
A procedure: a verb-noun, progressively-disclosed workflow. Nothing else is a skill.
Kernel
The deterministic runtime: gate engine, bp-yaml, release, eval harness. Tools, not prose.
Pack
Optional, independently-versioned bundle of skills + templates. Shipped via marketplace.
Cycle
One pass of the loop at any scope — the unit the ledger and metrics attach to.
The discipline
Rules → constitution. Tools → kernel. Docs → one file. Only procedures are skills.
The artifact model (B8, expanded)

One envelope wraps every file. One validator gates them all.

Today's bigpowers runs three overlapping doc conventions. bigspec collapses them into a single OKF envelope — every generated file carries the same frontmatter, so validate-okf is the gate for the whole specs/ tree.

# the universal envelope — every artifact starts here okf_kind: <string> # "story", "epic", "cockpit-state"… okf_version: "1.0" generated_by: <string> # "skill:elaborate-spec" | "human" generated_at: <iso-8601> supersedes: <filepath|null> # kills the _LATEST hack commit_range: <string|null> # traceability
cockpit-state
Machine state — active epic/story, current movement, handoff.next_skill. The "real API" from day 0.
read/written by the kernel
story
Durable intent + input to the counter. Context, actors, elements, NFRs, Gherkin. No gestalt SIZE field.
generated_by · elaborate-spec
glossary
Domain ground truth — entities, roles, quality rules. Input to native count-bcp (dims 5/6/7/11–13).
generated_by · define-language
epic
Roadmap container. WSJF prioritization + a story table that sums their BCP into a Job Size.
generated_by · plan-work
tasks
Execution ledger derived from acceptance criteria. Checkpoints transition FAILING → PASSING → VERIFIED.
generated_by · plan-work
tool-owned
Kinds the methodology references but never hand-templates — the producing tool owns the schema.
bcp-count · verification-report · story-metrics

Gate on okf_kind and structure — never on a value. A rule you can only verify by hard-coding a magic number is a rule in the wrong place.

How it runs

One loop, five movements, wrapped around the spec.

The same five movements run at project scale (once), epic scale (per epic), and story scale (per story). The spec is the fixed point every movement reads and refines — you never leave it behind a phase gate.

◆ THE SPEC — the fixed point ◆

02

03

decompose, risk-tier, size, order
04

05
Prove

P3 · fast lane

Trivial change: quick-fix → verify → ship. Gate discipline without gate fatigue.

P0–P2 · full lane

Risky change: the whole movement chain — spec, plan, TDD, review, security, completeness critic.

How it gets built

It builds itself — kernel-first, MCP-last.

The reborn repo bootstraps with a minimal seed of itself, then dogfoods the loop to build the rest. The deliberate inverse of bigpowers, which grew skills first and extracted infrastructure late — the source of most of its debt.

How it proves it's working

BCP runs the roadmap. Telemetry proves the work is real.

In bigpowers this lived in cycle-times.yaml and a dashboard — per-story BCP and BCP/hr velocity. bigspec keeps BCP as the headline and adds a thin quality ledger written per cycle, so two different questions both get real answers: "how much effort is left?" and "is it actually done?"

◆ BCP burn-down — effort remaining illustrative shape · real receipts published from M1

Sum the open stories' BCP → effort still to build. BCP-size a candidate epic before committing to it. Every count is gated on calibration_id, never the number — and velocity series are segmented by it, because a chart mixing calibrations is a silent lie.

What you get at the end

Two things ship. They never entangle again.

bigpowers conflated the methodology's own source with the artifact it drops into your project. bigspec separates them cleanly from the first commit.

Deliverable 1

The plugin

Kernel + ~29 core skills, installed once, globally. Authored in the portable Agent Skill format — no bespoke sync pipeline. Ships on a marketplace.

# install once, use everywhere
bigspec install
Deliverable 2

The scaffold

Per-project spec + cockpit templates, seeded into a project on init. Plain files: constitution template, empty cockpit, spec templates, agent config.

# seed a new project
bigspec init

The end state: the loop builds a real project cold, on plain files, and the evals prove each skill earns its place. A methodology that doesn't just have opinions — it has receipts.
